# Base System Status Matrix

This matrix summarizes the implemented base-system modules and their explicit boundaries. It is a handoff-oriented view; detailed implementation history remains in `docs/implementation_plan.md`, and incomplete or reserved work remains authoritative in `docs/known_gaps.md`.

Status terms:

- Complete: implemented for the confirmed base-system scope with tests or verification coverage.
- Partial: implemented with an explicit boundary documented in `docs/known_gaps.md`.
- Reserved: intentionally not implemented in v1 or requires a future confirmed goal.
- N/A: not applicable for that module.

## Cross-Cutting Release Boundaries

- Node.js is the only backend runtime target.
- SQLite remains usable for local development, local testing, and demo.
- PostgreSQL remains the supported deployment database.
- No SQL Server runtime, migration, or deployment implementation exists in v1.
- No example business module belongs in the base system.
- Redis, RabbitMQ, S3-compatible storage, SMS sending, real outbound webhook delivery, and a separate dead-letter queue remain optional or reserved unless a future confirmed goal implements them.
- API JSON IDs must remain serialized as strings.
- UTC timestamps and soft-delete fields must remain preserved where lifecycle deletion applies.
- Organization tree behavior must preserve the confirmed BIGINT materialized-path design.

## How To Use This Matrix

- For release or handoff: read this file with `docs/release_readiness.md` and the latest record under `docs/release_readiness_records/`.
- For local validation: follow `docs/local_run_acceptance.md`.
- For deployment validation: follow `docs/deployment_acceptance.md`.
- For incomplete or reserved work: treat `docs/known_gaps.md` as the authoritative source.
